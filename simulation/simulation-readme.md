# Peer2Nodes Simulation — Usage Guide

## Overview

The simulation lets you create multiple Peer2Nodes instances in a browser, open
authenticated encrypted channels between them, and exchange messages — all without any
native runtime or server. It exercises the full `PeerChannelManager` lifecycle:
ECDH key exchange, mutual challenge-response authentication, AES-256-GCM messaging,
and ACK tracking.

---

## Running the simulation

### Option A — Chrome / Edge (recommended, no server needed)

Open `simulation/index.html` directly from the filesystem:

```
# From the repo root
start simulation\index.html          # Windows
open simulation/index.html           # macOS
xdg-open simulation/index.html       # Linux
```

Chrome and Edge allow ES module imports from `file://` URIs. The page loads instantly
with two pre-created instances.

### Option B — Any browser (requires a local HTTP server)

Firefox and Safari block ES module imports from `file://` for security reasons.
Serve the directory over HTTP instead:

```bash
# Node.js (npx, no install needed)
npx serve simulation

# Python 3
python -m http.server 8080 --directory simulation

# Python 2
cd simulation && python -m SimpleHTTPServer 8080
```

Then open `http://localhost:8080` (or the port shown by the server).

### Browser requirements

| Browser | Minimum version | Notes |
|---|---|---|
| Chrome / Edge | 92+ | Works from `file://` |
| Firefox | 95+ | Requires HTTP server |
| Safari | 15.4+ | Requires HTTP server |

All crypto operations use the built-in `crypto.subtle` Web Crypto API — no extensions
or flags required on supported versions.

## Step-by-step testing

### 1. Create instances

On page load, two instances (`Node-1`, `Node-2`) are created automatically.

To create more:

- Type a name in the **Instance name…** input and press **Enter** or click **+ New**.
- Leave the field empty to use an auto-generated name (`Node-3`, `Node-4`, …).

Each instance gets its own P-256 identity keypair (generated by `PeerCryptoService`).

### 2. Open a channel

1. In the **CONNECT** bar, select an **Initiator** from the left dropdown.
2. Select a **Responder** from the right dropdown (must be a different instance).
3. Click **Open Channel**.

The initiator sends a HELLO and then initiates the 3-message mutual authentication
handshake. The full sequence completes in under 100 ms on any modern device:

```
Initiator → Responder : HELLO
Responder → Initiator : HELLO_ACK
Initiator → Responder : DATA / key-exchange   { ephemeralPubKey, identityKey, challenge }
Responder → Initiator : DATA / auth-response  { ephemeralPubKey, identityKey, challenge, sig }
Initiator → Responder : DATA / auth-confirm   { sig }
```

The channel card appears with status **READY** and a compose row.

### 2.5 Share connection info (QR + NFC)

1. Select an instance in **SHARE CONNECTION INFO**.
2. Click **Generate NFC + QR**.
3. Share using:
   - **Copy** / **Native Share** (shares the URI text),
   - **Render QR (remote)** to generate a QR image (opt-in; sends payload to `api.qrserver.com`),
   - **NFC Write** (writes URI to an NFC tag on supported mobile browsers).
4. To consume shared info:
   - paste a shared URI and click **Connect from Shared URI**, or
   - tap **NFC Read** then tap an NFC tag to auto-connect.

> **Web NFC note:** available on mobile Chromium browsers in secure contexts (HTTPS).
> When unavailable, use Copy/Native Share + QR workflow.

> **Note:** Both the initiator and the responder log `channel READY` independently —
> this reflects both sides completing mutual authentication, not a duplicate event.

### 3. Send messages

1. In the channel card's compose row, select the **sending instance** from the dropdown
   (defaults to the initiator; switch to send from the responder side).
2. Type your message and press **Enter** or click **Send**.

Each message is:
- Encrypted with AES-256-GCM using the session key derived during the handshake.
- Sent with `requireAck: true` (default) — the sender waits for an ACK from the receiver.
- Displayed in the channel's message history with a `✓` (sent) → `✓✓` (acknowledged) indicator.

The event log records both `sent` (sender side) and `received` (receiver side) events.

### 4. Open multiple channels

Multiple channels can be open simultaneously. Repeat step 2 with different instance
pairs. Each channel is independent and uses a distinct session key.

Example: with 3 instances (A, B, C) you can have A↔B and A↔C open at the same time.

### 5. Close a channel

Click the **✕** button in the top-right corner of a channel card. The initiator sends
a GOODBYE frame; both sides log `channel closed` and the card disappears.

## Observable protocol events (event log reference)

| Log event | Meaning |
|---|---|
| `instance created` | PeerCryptoService initialised; identity keypair generated; transport registered on the bus |
| `awaiting handshake` | Responder logged the incoming channel request |
| `opening channel →` | Initiator sent HELLO + KEY_EXCHANGE |
| `channel READY` | `PeerChannelManager.onChannelReady` fired — mutual auth complete, session key derived |
| `handshake complete` | `openChannel()` Promise resolved; sessionId confirmed in app.js |
| `sent "…"` | `sendMessage()` called; ciphertext transmitted |
| `received "…"` | `onMessageReceived` fired; plaintext decrypted and verified |
| `ACK` | `onMessageAcknowledged` fired; outbound message confirmed |
| `channel closed` | GOODBYE received or `closeChannel()` called |
| `error` | Auth failure, decrypt failure, or ACK timeout |

## Parameters and tuning

The following values can be changed directly in source code for testing edge cases.

### OutboundMessageQueue retry settings

File: `peer-channel-browser.js`, `PeerChannelManager` constructor:

```javascript
this.#queue = new OutboundMessageQueue({
  maxRetries:      3,      // retries before declaring a message expired
  retryIntervalMs: 5000,   // milliseconds between retry attempts
  onExpired: (id) => this.onChannelError?.(`ack_timeout:${id}`),
});
```

Set `retryIntervalMs: 1000` for faster retry visibility in the simulation.

### Forcing an auth failure

To test the `auth_failed` path, open the browser console and monkey-patch
`PeerCryptoService.prototype.verifyChallengeSignature` to return `false`:

```javascript
// Paste in the browser console before clicking "Open Channel"
const mod = await import('./peer-channel-browser.js');
mod.PeerCryptoService.prototype.verifyChallengeSignature = async () => false;
```

The initiator will log `error auth_failed:invalid_challenge_response` and the channel
will enter ERROR state.

### Instance name prefix

File: `app.js`:

```javascript
const name = nameInput.value.trim() || `Node-${nextNum}`;
```

Change `Node-` to any prefix.

### Auto-created instances on load

File: `app.js` (last two lines):

```javascript
createInstance().then(() => createInstance());
```

Remove this line to start with an empty instance list, or chain more calls to
pre-populate more instances.

### Instance color palette

File: `app.js`:

```javascript
const COLORS = ['#58a6ff', '#3fb950', '#f78166', '#d2a8ff', '#ffa657', '#79c0ff', '#ff7b72', '#56d364'];
```

Eight colors cycle in order. Add or reorder entries to taste.
